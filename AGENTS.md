# AGENTS.md

- 每次开始处理本仓库任务前，都要先阅读本文件。
- 每次与用户沟通时，都要加上用户称呼“李大王”。
- 不要在没有明确需求的情况下猜测业务逻辑。
- 不要随意添加字段。
- 涉及整体架构设计时，包括大的文件夹和每一个文件，都要写清楚作用等信息；能重复使用的功能和组件要尽可能复用，避免为相同功能写多份重复代码。
- 涉及 Flutter App、页面、组件、路由、主题、状态管理、前端服务、前端模型等开发时，必须同时阅读 `docs/frontend-app-architecture.md`。
- 可跨项目或跨模块复用的协作规则、工程规则、命名规则和维护规则放在本文件；具体项目 App 架构、文件职责、核心方法、可复用 widget 和用法记录在 `docs/frontend-app-architecture.md`。
- 新增或修改 App 目录结构、核心文件、公共方法、公共组件、页面路由、状态管理方式后，必须同步更新 `docs/frontend-app-architecture.md`，保证后续能按文档找到并复用。
- 不确定是否复用的代码，先放在具体业务目录；出现第二处真实复用需求时，再抽到 `shared/` 并补充文档。
- 已在项目文档中登记的公共工具、公共组件、公共方法必须优先复用；如果确实需要弃用或替换，必须先更新对应文档，写清楚替换原因和新用法。
- Flutter App 的 API 调用文件统一放在 `frontend/lib/api/`，数据模型统一放在 `frontend/lib/models/`；不要把 API 和 model 分散到各个模块目录中。
- Flutter App 后续新增或修改 API、model、组件、widget、公共功能方法、核心业务方法时，必须写最基本的备注信息，说明用途、输入输出、对应接口或使用场景，保证阅读代码时不费劲；备注要有信息量，避免“设置值”“返回数据”这类无意义注释。
- 工具类、封装方法、Controller 中声明的每一个方法都必须有基本注释；至少说清楚这个方法负责什么、何时使用，必要时说明输入输出和异常处理。
